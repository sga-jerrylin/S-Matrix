# E-order-20260422-1523-007-运行时诊断与审计能力落地指令单

## 1. 指令主题

把 Doris 4.0.5 和 Vanna 2 升级后的 runtime-facing 新能力落到 CLI/MCP/运行时层，重点是：

1. diagnosis
2. observability
3. audit-facing surfaces

## 2. 为什么由你负责

这些能力属于运行时壳层和运维可观测面，不属于查询业务实现。

你已有上下文连续：

1. `dc` CLI
2. MCP/runtime
3. `doctor` / `smoke`
4. Docker 联调环境与运行时健康检查

## 3. 本次要落实的“升级后新能力”

### A. Doris 4.0.5 诊断面落地

利用 Doris 4.0.5 更稳定的 profile / metrics / backend 状态，把 runtime 诊断从“活着没”升级成“为什么慢/为什么失败”。

要求：

1. 新增至少一条可用的 runtime diagnosis 能力
2. 诊断结果至少覆盖：
   - backend alive / health
   - recent query failure reason
   - system table write readiness
3. 保持 CLI-first，不要求前端先做

### B. Audit-facing runtime surface

利用 Vanna 2 的 audit / observability 思路，在 runtime 层先搭消费面。

要求：

1. `dc` / MCP 侧能读取或展示 query trace / context / audit-like metadata
2. 不要求本轮先做完整持久化审计系统
3. 先把“可见、可消费、可扩展”的壳层打出来

### C. MCP/CLI capability exposure

新增能力要能被：

1. CLI 调用
2. MCP tool 调用
3. 保持不重写业务逻辑，只代理现有稳定契约

## 4. 本次明确不做什么

1. 不改前端工作台
2. 不改查询主业务逻辑
3. 不搞第二套审计数据库
4. 不改 forecast 算法

## 5. 依赖关系

默认吃：

1. `agent-B` 输出的 query trace / context contract
2. `agent-A` 输出的 foundation capability / blocking reason

如果上游未全部交付：

1. 先落 runtime 壳层结构
2. 明确哪些字段是 conditional capability

## 6. 验收标准

必须全部满足：

1. 新增至少一个 CLI diagnosis 能力
2. 对应 MCP tool 也可调用
3. runtime 能展示 query trace / audit-like metadata
4. 不分叉第二套业务实现
