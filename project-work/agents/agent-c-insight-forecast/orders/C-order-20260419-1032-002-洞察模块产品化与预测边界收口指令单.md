## 你要做什么

- 把当前已有的 `analysis / expert` 能力收成正式的“洞察模块”，不要再停留在“已有代码能跑”状态。
- 收口 `/api/analysis/*` 相关返回结构，给出稳定的报告读模型和摘要字段。
- 明确 replay 分析、报告列表、报告详情、报告摘要之间的契约关系。
- 为 forecast 预留正式接口边界，但本轮不实现预测算法。
- 为 `SGA-Web` / `SGA-EastFactory` / `Evole` 预留协作接口草案，但本轮不做深接。

## 你不要做什么

- 不重写查询主链路。
- 不直接实现贝叶斯优化、时间序列或 `Evole` 算子接入。
- 不直接深接外部项目仓库。
- 不越权修改前端工作台和 CLI 壳层业务逻辑。
- 不为了“看起来完整”而提前定义一大套虚假的 forecast schema。

## 本轮必须收口的点

1. 洞察模块主契约
- 表分析请求与返回
- query-history replay 分析请求与返回
- 报告列表与详情的稳定读取面
- 报告摘要字段，供前端和 CLI 直接消费

2. 洞察结果结构
- `summary`
- `insights`
- `top_insights`
- `anomalies`
- `recommendations`
- `action_items`
- 统计摘要字段，如 `insight_count` / `anomaly_count`

3. 预测边界
- forecast 输入来自哪些内部指标
- forecast 未来允许接哪些外部信号
- forecast 结果最小输出结构应该长什么样
- 这轮只定义抽象接口，不落算法实现

4. 外部协作边界
- `SGA-Web` 负责什么外部信息补充
- `SGA-EastFactory` 负责什么趋势/舆情/外生信号补充
- `Evole` 如果未来接入，应该站在哪一层

## Sub-agent 授权

- 允许在 `analyst_agent.py`、`analysis_scheduler.py`、`analysis_dispatcher.py`、`main.py` 的分析相关接口范围内使用 sub-agent。
- 允许 sub-agent 分担“报告 schema 归一化”和“测试补强”两块工作。
- 不允许 sub-agent 越权修改查询主链路、前端、CLI/MCP 壳层。
