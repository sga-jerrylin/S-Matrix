## 查询链路说明

1. `NLQueryRequest` 接收 `query/table_names/resource_name/api_key/model/base_url/include_trace/max_repair_attempts`。
2. `PlannerAgent.plan()` 输出 `intent/tables/subtasks/needs_join/candidates/fallback_used/routing_reason`。
3. `TableAdminAgent.generate_sql_for_subtask_with_trace()` 为每个 subtask 生成 SQL，并产出 `strategy/prompt_attempts/schema_column_count/example_count/target_only/referenced_tables`。
4. `CoordinatorAgent.coordinate_with_trace()` 根据 relationship 选择 `passthrough / relationship_join / fallback_first_sql`。
5. 执行失败时，`RepairAgent.repair_sql_with_trace()` 输出按次记录的修复信息，主链路聚合为 `repair.attempts[]`。
6. 成功执行后写入 query history，并在响应中返回 `history_id` 与 `execution` 证据。

## 固定 Schema

- Request：
  `query`, `table_names[]`, `resource_name`, `api_key`, `model`, `base_url`, `include_trace`, `max_repair_attempts`
- Legacy compatibility：
  `question -> query`，`table_name -> table_names[0]`
- Response：
  `success`, `schema_version`, `query`, `intent`, `table_names[]`, `sql`, `data[]`, `count`, `history_id`, `warnings[]`, `trace`
- Trace sections：
  `planner`, `subtasks[]`, `orchestration`, `repair`, `execution`
