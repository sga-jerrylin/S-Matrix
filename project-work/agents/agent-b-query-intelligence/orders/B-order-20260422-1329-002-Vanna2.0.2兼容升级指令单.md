# B-order-20260422-1329-002-Vanna2.0.2兼容升级指令单

## 1. 指令主题

将查询智能层使用的 `vanna` 从 `0.7.9` 升级到 `2.0.2`，但本轮采用**兼容升级策略**，不强行重写到 Vanna 2 Agent/Web-first 架构。

## 2. 为什么由你负责

Vanna 当前属于查询智能链路的一部分，不属于运行时 Docker 层。

你负责的上下文天然覆盖：

1. `requirements.txt`
2. `vanna_doris.py`
3. `main.py` 中自然语言查询链路
4. 查询智能相关测试

所以这轮由 `agent-B` 做，不新开 agent。

## 3. 升级策略

这次不是“全面吃 Vanna 2 新架构”，而是：

1. 包版本升级到 `vanna==2.0.2`
2. 保持当前 DorisClaw 查询能力和接口契约不变
3. 优先走 **legacy compatibility path**

你必须基于官方 Vanna 2 迁移路径来做：

1. 允许使用 `vanna.legacy.*`
2. 允许引入本地 compatibility shim
3. 不要求本轮直接接入 `<vanna-chat>`、FastAPI server、Agent UI

结论：

本轮目标是“先跑在 2.0.2 包上”，不是“把 DorisClaw 重构成 Vanna 官方示例”。

## 4. 本次要做什么

### A. 依赖升级

1. `vanna==0.7.9` -> `vanna==2.0.2`
2. 若需要补充与 2.0.2 兼容的最小依赖，必须说明原因

### B. 兼容接入

建议优先路径：

1. 引入本地 `vanna_compat`（或等价兼容层）
2. 把现有 `from vanna.base import VannaBase` 之类的旧路径兼容到 `vanna.legacy.base`
3. 保持 `VannaDoris` / `VannaDorisOpenAI` 语义不变

### C. 查询链路回归

至少要覆盖：

1. 自然语言查询主链路
2. DDL / documentation / history retrieval
3. `add_question_sql` / `get_similar_question_sql`
4. 相关 contract 测试

## 5. 本次明确不做什么

1. 不重写成 Vanna 2 Agent API 主架构
2. 不接 `<vanna-chat>` Web component
3. 不改前端工作台来适配 Vanna 官方 UI
4. 不改 CLI/MCP 业务契约
5. 不改 Doris runtime / Docker 脚本

## 6. 与其他 agent 的关系

你和 `agent-E` 在同一波升级里，但不要混写 owner 文件。

要求：

1. 不碰 `docker-compose.yml` / `init.*` / `update.*` / `doctor.py`
2. 若你需要做动态联调，默认基于 `agent-E` 已交付并通过的 Doris 4.0.5 运行时
3. 你的最终动态验证以 `agent-E` runtime green 为前提
