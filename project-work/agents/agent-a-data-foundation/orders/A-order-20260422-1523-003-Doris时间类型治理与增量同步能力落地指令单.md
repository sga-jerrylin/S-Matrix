# A-order-20260422-1523-003-Doris时间类型治理与增量同步能力落地指令单

## 1. 指令主题

把 Doris 4.0.x 升级后真正能落地的数据基础能力收口到 foundation 层，重点是：

1. 时间类型治理
2. 指标时间序列语义收紧
3. 增量同步能力落地

## 2. 为什么由你负责

这些能力都落在 data foundation 边界内，与你当前 owner 的上下文连续：

1. source state / registry / metadata / relationship
2. metric foundation / availability / series
3. foundation internal/external contract

## 3. 本次要落实的“升级后新能力”

本次不是泛泛谈 Doris 4，而是只落 3 个和 DorisClaw 当前主线直接相关的点：

### A. 类型安全时间过滤

把当前 foundation / metric 里的字符串式时间比较，收口为 Doris 4 的类型安全时间表达式。

要求：

1. 不再依赖“源字段格式刚好一致”的字符串比较
2. 对时间字段使用显式 cast / try_cast / 标准化表达式
3. day / week / month 粒度统一走一套时间桶逻辑

### B. 指标时间序列语义收紧

利用 Doris 4.x 的类型处理和时间函数能力，把指标 series 的可解释性收紧。

要求：

1. metric series 返回中明确时间字段规范化结果
2. 对非法时间字段、不可解析时间值给出稳定 blocking reason
3. foundation metric read contract 保持稳定，不破坏现有 forecast 消费面

### C. 增量同步 MVP

把“升级后的 Doris 能力”落实到同步层，而不是只停留在版本说明。

本轮目标不是做一个万能 CDC 系统，而是落一个最小可用的增量同步 MVP：

1. 在当前 datasource sync 路径中引入 `full` / `incremental` 策略区分
2. 至少支持一种稳定增量路径：
   - 基于时间列的 append/incremental 同步
3. 对不满足条件的表，必须明确回退到 full sync，并给出 capability/explanation

## 4. 本次明确不做什么

1. 不改查询智能层
2. 不改前端工作台
3. 不改 CLI/MCP/runtime
4. 不做复杂 CDC / binlog / Debezium 方案
5. 不改预测算法

## 5. 交付要求

必须包含：

1. foundation contract 变更点
2. 增量同步 capability 说明
3. blocking reason 设计
4. 回归测试

## 6. 验收标准

必须全部满足：

1. metric series 不再依赖脆弱字符串时间比较
2. day/week/month 粒度结果稳定
3. 增量同步 MVP 可运行
4. 不可增量的表会明确说明原因并回退 full sync
5. foundation / internal contract 不破坏当前 forecast 与 query 依赖

## 7. 返回格式

1. 改动文件清单
2. Doris 4 能力具体落了什么
3. foundation contract 变化
4. 回归命令与结果
5. 风险说明
