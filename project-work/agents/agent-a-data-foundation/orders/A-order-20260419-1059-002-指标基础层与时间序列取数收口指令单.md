## 你要做什么

- 在数据基础层之上，补出“预测可用”的指标基础层。
- 让 `metric_key` 不再只是一个字符串，而是可被正式读取、校验、取数和解释的指标定义。
- 收口时间序列取数面，明确哪些表、哪些时间字段、哪些聚合方法可以进入预测。
- 给 `agent-c` 提供稳定的内部指标读取面。

## 你不要做什么

- 不改查询智能主链路。
- 不做预测算法。
- 不直接接入外部协作数据源。
- 不越权修改前端、CLI、runtime 壳层。
- 不为了“通用”而做一整套过度设计的指标平台。

## 本轮必须收口的点

1. 指标定义
- `metric_key`
- `display_name`
- `description`
- `table_name`
- `time_field`
- `value_field` 或聚合表达式
- `aggregation`
- `default_grain`
- `dimensions`

2. 时间序列取数面
- 给出统一读取接口或服务方法
- 输入：`metric_key`、时间范围、粒度、可选维度过滤
- 输出：按时间排序的点列

3. 指标可用性边界
- 哪些指标现在允许进入 forecast MVP
- 哪些表暂时不允许进入 forecast
- 哪些缺少时间字段或口径不稳定的指标必须排除

- `doris-api/datasource_handler.py`

## Sub-agent 授权

- 允许在数据基础层和指标读取面范围内使用 sub-agent。
- 允许 sub-agent 分担“指标 schema 归一化”和“测试补强”。
- 不允许 sub-agent 越权进入预测算法、前端、CLI 层。
