## 当前判断

第一轮基建的主阻塞项已经基本清掉，可以进入第二轮。

已形成可继续推进的基础：
- `agent-a` 已把数据基础层收口到稳定读模型与统一入库链路。
- `agent-b` 已把查询智能层收口到可回归状态，现阶段可以作为洞察层输入来源。

- 前端已经有分析页面基础，但还不适合在 forecast 未定时先卷复杂交互。
- CLI 壳层已经有占位能力，应该等待洞察 / 报告 contract 进一步稳定后再继续扩面。

## 本轮启动策略

本轮只正式启动两个方向：

1. `agent-c`
目标：把“已有 analysis / expert 能力”收口成正式洞察模块，并为后续 forecast 与外部协作预留清晰接口。

2. `agent-d`
目标：不是直接冲 forecast 页面，而是先把工作台收成“上传 -> 查询 -> 洞察”三段主路径，并对齐现有稳定 schema。

本轮暂不新增工单的 agent：
- `agent-a`
说明：转入支持态，只处理数据基础层契约澄清或被动修复，不主动扩面。
- `agent-b`
说明：转入稳定态，不再主动改查询主链路；如 `agent-c` 发现 query evidence / replay contract 缺口，再按边界开补单。
- `agent-e`
说明：转入封板态；`dc report / dc forecast` 继续保持占位，等待 `agent-c` 先把洞察/报告 contract 收稳。

## 本轮目标

本轮结束时，项目至少应达到下面三个状态：

1. 后端已经有可被消费的正式洞察 contract
包括但不限于：
- 表分析
- 查询历史 replay 分析
- 报告读取
- 报告列表
- 报告摘要字段

2. 前端工作台已经明确围绕三条主链路收口
- 上传
- 查询
- 洞察

3. forecast 仍然不急着实现，但要把接口边界说清楚
- 什么是内部指标输入
- 什么是外部协作输入
- 什么是预测结果输出

## 本轮验收口径

- 不以“又加了多少功能”为标准。
- 以“模块边界是否更清楚、接口是否更稳定、后续 forecast 是否更容易接入”为标准。

## 风险提醒

- `agent-c` 如果把 forecast 和外部协作一起直接做深，很容易失控。
- `agent-d` 如果在 schema 未收口前做太重的前端结构重写，返工概率会很高。
- `agent-e` 现在不该抢先定义 `report / forecast` 输出，否则会反向绑死后端。
