# 第三轮推进方案

- 时间：`2026-04-19 10:59:25 +08:00`
- 阶段：`Batch 3`
- 项目：`DorisClaw / DC`

## 当前状态

到这一轮开始前，项目已经形成下面这条可继续推进的基线：

- `agent-a` 已收口数据基础层，统一了入库链路、稳定读模型与关系读取面。
- `agent-b` 已把查询智能层收口到可回归状态，可作为洞察与预测的上游输入。
- `agent-c` 已把洞察模块 contract 收口，并补出了 forecast / collaboration 的边界草案。
- `agent-d` 已把前端工作台收成上传 / 查询 / 洞察 / 预测占位四段主路径。
- `agent-e` 已把 CLI / MCP / runtime 收到同一套壳层，并补齐旧 MCP 最小兼容闭环。

这意味着下一步不该再继续做“边界草案”，而是要把 `forecast` 从 `contract_only` 推到“最小可运行闭环”。

## 本轮目标

本轮核心目标只有一个：

把 `forecast` 从占位能力推进成内部可跑的 MVP。

但这个目标不能由一个 agent 单独完成，必须拆成三段连续链路：

1. `agent-a`
做指标基础层，解决“预测到底预测什么、从哪里取数、时间粒度怎么定义”。

2. `agent-c`
在指标基础层之上做预测 MVP，先只做内部指标，不做外部信号融合，不做优化算子。

3. `agent-e`
把已经稳定的 insight / forecast 能力真正挂到 `dc` CLI 与 runtime 消费面，形成对 `SGA-Molt` 可调用的能力出口。

## 本轮为什么这样拆

这轮不能直接让 `agent-c` 单独上预测算法，原因很明确：

- 没有指标基础层，`metric_key` 只是字符串，不是能力。
- 没有 CLI / runtime 消费面，预测就算跑出来也还没有形成对外可用能力。
- 过早引入 `SGA-Web` / `SGA-EastFactory` / `Evole`，只会把 MVP 拉散。

所以这轮的策略是：

- 先做内部指标闭环。
- 先做最小预测闭环。
- 先做可调用闭环。
- 暂不做外部协作深接。

## 本轮分工

### `agent-a`

职责：
- 指标基础层
- 时间序列取数面
- 指标元信息与聚合口径

本轮重点：
- 明确 `metric_key`
- 明确时间字段
- 明确粒度与聚合方法
- 输出稳定的指标读取面，供 `agent-c` 使用

### `agent-c`

职责：
- 预测 MVP
- 回测摘要
- 预测结果 contract

本轮重点：
- 只做内部指标预测
- 返回预测点列、区间、假设、回测摘要
- 不接外部仓库、不接优化器

### `agent-e`

职责：
- `dc report`
- `dc forecast`
- runtime / MCP 消费面接入

本轮重点：
- `dc report` 从占位变成真实读取能力
- `dc forecast` 跟随 `agent-c` 的 MVP contract 挂接
- 不重写业务逻辑，只封装稳定能力面

## 本轮不启动的模块

- `agent-b`
说明：查询智能层先保持稳定，只有当 `agent-c` 发现 query evidence / replay contract 存在明确缺口时，再单独补单。

- `agent-d`
说明：前端已有预测入口占位，这轮不继续大动。等 `forecast` 输出稳定后，再开新一轮前端接入。

## 本轮验收口径

本轮通过标准不是“预测做得多复杂”，而是下面三点：

1. 存在真实可取数的 `metric_key`
2. `/api/analysis/forecast` 不再只是 `contract_only`
3. `dc forecast` 能实际消费这个输出

## 风险提醒

- 如果 `agent-a` 直接做“指标宇宙”，本轮会失控。
- 如果 `agent-c` 直接上贝叶斯优化 / 外部信号融合，本轮会失控。
- 如果 `agent-e` 抢先定义输出结构，本轮会反向绑死后端。
