# 产品分层规划

## 1. 总体判断

`DorisClaw` 后期不应只被理解为“自然语言查数工具”。

它更合理的产品分层应该是：

1. 查询层
2. 洞察层
3. 预测层
4. 决策编排层

其中，外部信息和外部趋势属于协同能力，不是本项目独立重做的底层。

## 2. 查询层

回答的问题：

- 现在发生了什么
- 哪些数据异常
- 哪些维度在变化

核心能力：

- 自然语言转 SQL
- 表关系理解
- 多表查询
- 查询修复
- 查询记忆与结果复用

这一层的要求是：

- 准
- 稳
- 可解释

## 3. 洞察层

回答的问题：

- 为什么会这样
- 哪些因素最可能驱动了变化
- 现象背后还有哪些异常、拐点、群组差异

我建议把“数据深挖”定义成两段式：

### 3.1 内部深挖

- 指标拆解
- 维度贡献分析
- 异常点发现
- 时间段变化分析
- 根因候选生成

### 3.2 协同深挖

- 内部数据解释不够时，调用 `SGA-Web` 补外部背景
- 需要外部趋势判断时，调用 `SGA-EastFactory`
- 形成“内部结论 + 外部证据 + 外部趋势”的联合解释

所以“深挖”不是多跑几轮 SQL，而是协同式深挖。

## 4. 预测层

回答的问题：

- 接下来会怎样
- 未来值大概是多少
- 风险区间有多大
- 哪些因素最敏感

### 4.1 内部预测

`DorisClaw` 负责：

- 内部业务指标预测
- 回测
- 区间预测
- 外生变量接入
- 情景模拟

### 4.2 外部预测协同

`SGA-EastFactory` 负责：

- 外部情绪、热点、事件趋势的预测

`DorisClaw` 负责：

- 把这些外部结果接成自己的 exogenous features
- 用于提升内部业务指标预测效果

### 4.3 贝叶斯优化的角色

贝叶斯优化不应作为产品主角出现。

更合理的位置是：

- 用于模型参数搜索
- 用于窗口长度、特征组合、阈值优化
- 用于策略模拟和实验调优

也就是说，它是优化器，不是业务用户理解的预测引擎本体。

## 5. 决策编排层

回答的问题：

- 所以现在应该做什么
- 哪些行动最值得优先执行
- 哪些结论需要人确认

核心能力：

- 报告生成
- 关键结论摘要
- 动作建议
- 风险提示
- 人工确认点
- 可回放、可追溯的决策链

这一层是让查询、洞察和预测真正进入业务流程，而不是停在页面结果里。

## 6. 产品层与算法层的拆分建议

### `DorisClaw` 保留在产品层

它负责：

- 输入输出
- 任务编排
- 结果解释
- CLI / MCP 暴露
- 与其他项目的协同

### `Evole` 未来承接算法层

它负责：

- 模型演化
- 算法评估
- 超参优化
- 版本对比
- 优化策略管理

## 7. 推荐的落地顺序

### 第一阶段

- 把查询层做稳定
- 建立黄金查询集和验收标准

### 第二阶段

- 做内部深挖
- 定义异常、变化、根因候选的输出结构

### 第三阶段

- 接 `SGA-Web` 做协同深挖
- 让解释能力从“只看表”升级为“看业务场景”

### 第四阶段

- 上业务预测
- 先做时间序列预测、回测和区间

### 第五阶段

- 接 `SGA-EastFactory`
- 使用外部趋势作为外生变量

### 第六阶段

- 把优化与演化逐步抽给 `Evole`

## 8. 最小产品主线

如果只保留一条最小但完整的产品主线，我建议是：

1. 导入业务数据
2. 用自然语言查询
3. 对异常指标做一次深挖
4. 补充外部证据
5. 给出下一周期预测
6. 输出可读摘要和建议动作

这条链路比单纯“会查数”更像一个完整产品。
