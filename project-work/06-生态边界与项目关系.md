# 生态边界与项目关系

## 1. 命名与定位

- 本项目未来正式名称：`DorisClaw`
- 简称：`DC`
- 当前仓库路径仍为 `SGA-DorisClaw`
- 历史文档与旧阶段描述中出现的 `S-Matrix` 视为本项目旧名称

## 2. 生态中的角色

`DorisClaw` 应被定义为：

一个面向内部业务数据的查询、协同洞察与业务预测引擎。

它的重点不是替代整个 SGA 生态，而是在生态中承担“内部数据决策层”的职责。

## 3. 四个关联项目的边界

### `SGA-Web`

角色：

- 外部信息获取层
- 提供搜索、抓取、公众号导出、浏览器桥、CLI 能力

`DorisClaw` 对它的使用方式：

- 查询外部文章、网页正文、公众号内容、行业资讯
- 为内部数据深挖补充外部证据和背景
- 不吸收其抓取、浏览器、导出能力到本项目内部重复实现

### `SGA-EastFactory`

角色：

- 外部舆情与趋势预测层
- 面向外部世界的 sentiment/trend prediction service

`DorisClaw` 对它的使用方式：

- 当内部业务指标受外部情绪、热点、事件影响时，拉取其趋势预测结果
- 将其结果作为外生变量、解释变量或决策参考
- 不在 `DorisClaw` 内部重做完整外部舆情预测平台

### `Evole`

角色：

- 算法进化与优化层
- 承接模型迭代、超参搜索、评估和进化逻辑

`DorisClaw` 对它的使用方式：

- 先抽象预测与优化接口
- 后续把算法优化、模型演化和自动调优逐步外移到 `Evole`
- 不让产品项目被算法实验耦死

### `SGA-Molt`

角色：

- agent runtime / skills 执行与编排层

`DorisClaw` 对它的使用方式：

- 作为下游能力项目，被 `Molt` 的 skills 稳定调用
- 通过 CLI 和 MCP 暴露能力，而不是让 `Molt` 直接访问内部数据库逻辑

## 4. DorisClaw 自己应该拥有的职责

- 内部结构化数据的查询
- 内部数据异常、变化、原因候选的深挖
- 内部业务指标预测
- 外部信息和外部趋势的协同整合
- 可被上游系统消费的 CLI / MCP 能力面
- 人类可读的报告、摘要、解释和建议

## 5. DorisClaw 不应该拥有的职责

- 通用外网搜索底座
- 通用网页抓取与浏览器自动化平台
- 完整外部舆情预测平台
- 算法进化实验室
- 复杂上游编排运行时

## 6. 协同原则

原则不是“吸收”，而是“调用”。

也就是说：

- `DorisClaw` 调 `SGA-Web`
- `DorisClaw` 调 `SGA-EastFactory`
- `DorisClaw` 未来把优化任务交给 `Evole`
- `SGA-Molt` 通过 skills 调 `DorisClaw`

这样每个项目都能保持边界稳定。

## 7. 典型协同链路

### 链路 A：内部指标异常解释

1. `DorisClaw` 发现内部指标异常
2. `DorisClaw` 做内部数据深挖
3. 若内部解释不足，则调用 `SGA-Web` 获取外部证据
4. 汇总为“异常 + 候选原因 + 外部证据”的解释结果

### 链路 B：业务预测

1. `DorisClaw` 基于内部历史指标做时间序列预测
2. 若需要外部趋势输入，则调用 `SGA-EastFactory`
3. 将外部预测结果作为外生变量接入
4. 输出业务预测值、区间、主要驱动因素和建议动作

### 链路 C：上游技能调用

1. `SGA-Molt` skill 调用 `DorisClaw CLI`
2. `DorisClaw` 执行查询、洞察或预测
3. 必要时向 `SGA-Web` / `SGA-EastFactory` 发起协同请求
4. 最终以标准 JSON 结果返回给 `Molt`

## 8. 当前建议

当前优先级不是扩更多项目能力，而是先固定边界。

只要边界不清，三个项目就会开始长重复能力。
