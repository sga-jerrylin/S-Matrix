# Agent 协作与审核规范

## 1. 目的

这份规范用于把 `DorisClaw` 的 agent 协作方式从“口头分工”变成“有目录、有编号、有审核、有返工”的可执行机制。

## 2. 目录标准

每个 agent 必须有独立目录，当前统一放在 `project-work/agents/` 下。

标准结构如下：

```text
project-work/agents/<agent-folder>/
  README.md
  orders/
  tickets/
  reviews/
    passed/
    rework/
```

说明：

- `README.md`：该 agent 的模块边界、责任范围、当前批次
- `orders/`：项目经理发给该 agent 的指令单
- `tickets/`：该 agent 的正式工单
- `reviews/passed/`：审核通过记录
- `reviews/rework/`：返工单和驳回记录

## 3. 编号规则

所有文件必须同时带“编号”和“时间”。

统一命名规则：

```text
<agent-prefix>-<kind>-YYYYMMDD-HHmm-<seq>-<title>.md
```

当前约定：

- `agent-a` 前缀：`A`
- `agent-b` 前缀：`B`
- `agent-c` 前缀：`C`
- `agent-d` 前缀：`D`
- `agent-e` 前缀：`E`

文件类型：

- `order`：指令单
- `ticket`：工单
- `review`：审核单
- `rework`：返工单

例如：

- `A-order-20260419-0049-001-数据基础层基建启动指令单.md`
- `A-ticket-20260419-0049-001-数据基础层基建启动工单.md`
- `A-review-20260420-1030-001-数据基础层基建审核单.md`
- `A-rework-20260420-1045-001-数据基础层基建返工单.md`

## 4. 文件职责

### 指令单

指令单用于写清楚：

- 这次明确要做什么
- 这次明确不做什么
- 有哪些边界不能越过
- 是否授权使用 sub-agent
- 汇报时要交什么

### 工单

工单用于写清楚：

- 目标
- 范围
- 完成标准
- 交付物
- 依赖关系
- 风险

### 审核单

审核单用于给出正式结论：

- 通过
- 有条件通过
- 不通过

审核单必须写清楚依据，不能只写一句“感觉不行”。

### 返工单

返工单用于写清楚：

- 哪些点不通过
- 必须修到什么程度
- 返工范围是什么
- 再次提交时必须附带什么验证

## 5. 流程

标准流程只有五步：

1. 项目经理先发指令单
2. 再落正式工单
3. agent 执行并提交交付物
4. 项目经理审核并形成审核单
5. 不通过则发返工单，通过则归档到 `reviews/passed/`

## 6. Sub-agent 授权规则

每个 agent 默认允许在自己模块内部使用 sub-agent 提高效率，但必须遵守以下规则：

- 只能拆本模块内部的子任务
- 不能把跨模块设计决策外包给 sub-agent
- 不能修改不属于自己 owner 范围的主文件
- 不能绕过工单和审核流程
- sub-agent 的产出必须回填到主工单总结里
- 最终质量责任仍由对应 agent owner 承担

## 7. 审核标准

默认审核维度：

- 是否完成工单目标
- 是否遵守范围边界
- 是否引入跨模块污染
- 是否给出必要验证
- 是否留痕完整

满足以上条件才允许通过。

## 8. 当前生效范围

本规范从 `2026-04-19` 起用于第一轮基建的五个 agent：

- `agent-a`：数据基础层
- `agent-b`：查询智能层
- `agent-c`：洞察与预测层
- `agent-d`：前端工作台层
- `agent-e`：CLI / MCP / 运行时层
