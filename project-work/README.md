# DorisClaw 项目基础文件

`project-work/` 用来承接 `DorisClaw / DC` 的项目管理文件、agent 协作目录和过程留痕。

当前采用两层管理方式：

- 上层文档负责定义方向、边界、路线和批次
- `agents/` 目录负责承接每个 agent 的工单、指令单、审核单和返工单

## 项目命名

- 项目未来正式名称：`DorisClaw`
- 项目简称：`DC`
- 仓库与历史文档中仍可能出现 `S-Matrix`，当前视为历史名称或旧阶段称呼

## 我对这个项目的理解

从仓库现有代码和文档来看，`DorisClaw` 的核心方向是：

- 以 `Apache Doris` 作为分析型数据底座
- 支持 Excel 上传、建表、导入、查询和数据源同步
- 通过 `Vanna.AI + LLM` 做自然语言转 SQL
- 通过规划、生成、协调、修复等链路提升复杂查询成功率
- 提供一个前端控制台给用户做上传、查询、配置和分析查看
- 向后演进为“查询 + 协同洞察 + 业务预测”的数据决策能力层

换句话说，这不是一个通用聊天产品，而是一个“面向数据查询与分析”的智能数据中台原型。

## 现在的管理判断

虽然仓库里已经有不少代码和历史计划，但从正式项目管理角度，我们暂时把它视为：

- `项目启动中`
- `基础方向明确`
- `代码已有原型`
- `第一轮基建即将按模块 owner 开始执行`

所以这里的文档目标不是回溯历史，而是把项目启动所需的最小基础盘和第一轮执行盘一起搭起来。

## 当前目录说明

- `01-项目概述.md`：项目目标、范围、核心价值
- `02-当前状态.md`：当前阶段、已有资产、主要问题
- `03-路线图.md`：分阶段推进思路
- `04-任务清单.md`：当前待办和优先级
- `05-工作约定.md`：文档、任务、变更、agent 审核的基本规则
- `06-生态边界与项目关系.md`：和 `SGA-Web`、`SGA-EastFactory`、`Evole`、`SGA-Molt` 的边界定义
- `07-产品分层规划.md`：查询、洞察、预测、决策编排的产品分层
- `08-CLI与Molt集成草案.md`：CLI-first 与 skills/MCP 集成草案
- `09-第一轮基建分工方案.md`：第一轮基建的模块 owner 拆分与批次建议
- `10-Agent协作与审核规范.md`：agent 文件夹、工单编号、审核和返工流程
- `agents/`：每个 agent 的独立工作目录
- `templates/任务模板.md`：单个任务描述模板
- `templates/周报模板.md`：后续同步进展时使用
- `templates/Agent工单模板.md`：agent 工单模板
- `templates/Agent指令单模板.md`：agent 指令单模板
- `templates/Agent审核单模板.md`：agent 审核单模板
- `templates/Agent返工单模板.md`：agent 返工单模板

## 使用原则

- 项目概述负责定义方向，不写过细执行内容
- 当前状态负责讲事实，不写愿景话术
- 路线图负责排阶段，不替代任务拆解
- 任务清单负责落地执行，不堆长篇讨论
- agent 执行必须落文件，不能只留口头沟通
- agent 交付必须经过审核，不通过必须发返工单
