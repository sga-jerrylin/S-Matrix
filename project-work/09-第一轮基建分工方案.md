# 第一轮基建分工方案

## 1. 分工原则

这轮分工不按“人数平均”来拆，而按“模块连续性”来拆。

原则只有四条：

- 一个 agent 长期盯一个模块，不来回切主题
- 跨模块只通过接口和契约协同，不互相侵入实现
- 不为了把人用满而硬拆
- 当前先启用必要 agent，不强行全开

## 2. 当前建议启用的 agent

第一轮基建建议启用 5 条 owner 线，但可以按优先级分批启动：

- `agent-a`：数据基础层
- `agent-b`：查询智能层
- `agent-c`：洞察与预测层
- `agent-d`：前端工作台层
- `agent-e`：CLI / MCP / 运行时层

如果资源有限，启动顺序建议为：

1. `agent-a`
2. `agent-b`
3. `agent-e`
4. `agent-c`
5. `agent-d`

原因是第一轮基建最先要稳定的是底层、查询和能力输出，而不是先卷页面。

## 3. 模块划分

### `agent-a`：数据基础层

职责：

- 数据接入
- 表注册与元数据
- 数据源同步
- Doris 访问基础能力

建议持续负责的文件：

- `doris-api/db.py`
- `doris-api/upload_handler.py`
- `doris-api/datasource_handler.py`
- `doris-api/metadata_analyzer.py`
- `doris-api/handlers.py` 中与数据接入相关部分
- `doris-api/main.py` 中与上传、表管理、数据源相关接口

第一轮重点：

- 统一数据接入路径
- 明确表、字段、元数据、关系的基础模型
- 稳定上传、注册、同步、刷新链路

为什么这么拆：

- 这些能力都围绕“数据进入 Doris 以及被正确理解”展开
- 它们天然属于一个连续上下文，不应拆给多个人

### `agent-b`：查询智能层

职责：

- 自然语言查询
- SQL 生成与修复
- 检索记忆
- 查询编排

建议持续负责的文件：

- `doris-api/vanna_doris.py`
- `doris-api/planner_agent.py`
- `doris-api/table_admin_agent.py`
- `doris-api/coordinator_agent.py`
- `doris-api/repair_agent.py`
- `doris-api/embedding.py`
- `doris-api/main.py` 中与 `/api/query/*` 相关接口

第一轮重点：

- 把查询链路做成可解释、可验证、可回归
- 固定查询输入输出结构
- 形成 golden queries 和回归标准

为什么这么拆：

- 查询问题不是单个文件问题，而是一整条智能查询链路
- 同一个 owner 才能持续收敛效果，而不是今天修 planner、明天换人修 repair

### `agent-c`：洞察与预测层

职责：

- 数据深挖
- 分析报告
- 预测抽象层
- 对 `SGA-Web` / `SGA-EastFactory` / `Evole` 的协同接口

建议持续负责的文件：

- `doris-api/analyst_agent.py`
- `doris-api/analysis_scheduler.py`
- `doris-api/analysis_dispatcher.py`
- `doris-api/app_scheduler.py`
- `doris-api/main.py` 中与 `/api/analysis/*` 相关接口

第一轮重点：

- 把“分析报告”从已有代码能力收敛成正式产品能力
- 抽象内部预测接口，不急着把 `Evole` 真接进来
- 预留对 `SGA-Web` 和 `SGA-EastFactory` 的上下文/趋势协同接口

为什么这么拆：

- 洞察和预测本来就是一条连续能力线
- 如果把报告、预测、调度、外部协同拆散，后面会很难形成统一产品逻辑

### `agent-d`：前端工作台层

职责：

- 前端页面
- 交互链路
- 结果展示
- 查询、洞察、预测的可视工作台

建议持续负责的文件：

- `doris-frontend/src/App.vue`
- `doris-frontend/src/components/*`
- `doris-frontend/src/api/*`
- `doris-frontend/src/main.ts`
- `doris-frontend/src/style.css`

第一轮重点：

- 把上传、查询、分析三个主入口收口
- 把页面从“功能堆叠”整理成“任务导向”
- 准备后续的预测展示位置，但不要现在就过度设计

为什么这么拆：

- 前端需要长期保持页面结构和交互的一致认知
- 不适合被查询 agent、预测 agent 各自顺手改一点

### `agent-e`：CLI / MCP / 运行时层

职责：

- CLI-first 能力输出
- MCP server 暴露
- 启动脚本、健康检查、Docker、集成契约

建议持续负责的文件：

- `doris-api/mcp_server.py`
- `mcp_config.json`
- `scripts/*`
- 根目录 `docker-compose.yml`
- `init.ps1`, `init.sh`, `update.ps1`, `update.sh`
- 未来新增的 `dc` CLI 入口相关文件

第一轮重点：

- 把 `DorisClaw` 真正变成可被 `SGA-Molt` 调用的能力项目
- 先做 `health / query / insight / forecast / mcp serve` 的 CLI 骨架
- 把运行时、配置、健康检查、最小 smoke 流程稳定下来

为什么这么拆：

- CLI、MCP、Docker、脚本、健康检查本质上是一条“能力输出与运行时”主线
- 它和前端不是一回事，也不该零散挂给后端 owner 顺手维护

## 4. 当前不建议单独拆 agent 的模块

以下方向当前不建议独立拆成单独 agent：

- 文档 agent
- 测试 agent
- 集成 agent
- 外部协同 agent

原因：

- 文档应跟随模块 owner 更新
- 测试应跟随对应模块 owner 建立
- 外部协同目前还在接口定义阶段，还不值得单独拉一条线

## 5. 模块交界契约

### `agent-a` -> `agent-b`

交界内容：

- 表、字段、元数据、关系的标准读取接口

要求：

- 查询层不能直接依赖上传细节
- 只依赖稳定的 metadata / registry / relationship 输出

### `agent-b` -> `agent-c`

交界内容：

- 查询历史
- 查询结果摘要
- 查询执行证据

要求：

- 洞察层不重新发明查询主链路
- 复用查询层已有产出做 replay 和深挖

### `agent-b` / `agent-c` -> `agent-d`

交界内容：

- 查询结果 schema
- 分析报告 schema
- 预测结果 schema

要求：

- 先定返回结构，再做页面
- 前端不自行猜字段

### 全部模块 -> `agent-e`

交界内容：

- CLI 输入参数
- JSON 输出格式
- 健康检查与错误码
- MCP 工具定义

要求：

- `agent-e` 不重写业务逻辑
- 只封装稳定能力面

## 6. 第一轮基建建议任务包

### `agent-a`

- 收口上传、元数据、关系、数据源同步的基础模型
- 输出统一表语义读取接口

### `agent-b`

- 收口查询链路
- 建 golden query 集
- 固定查询结果 schema

### `agent-c`

- 把现有 analysis / expert 能力升级成正式洞察模块
- 定义 forecast 抽象接口
- 先做外部协同接口草案，不急着深接

### `agent-d`

- 前端只围绕三块做收口：
  - 上传
  - 查询
  - 洞察

### `agent-e`

- 起 `dc` CLI 骨架
- 起 MCP 服务骨架
- 稳定 Docker / health / smoke

## 7. 分批启动建议

### Batch 1

- `agent-a`
- `agent-b`
- `agent-e`

目标：

- 先把“数据进得来、查得出来、能力能被调用”打稳

### Batch 2

- `agent-c`

目标：

- 把分析与预测能力从“已有代码”收成“正式能力”

### Batch 3

- `agent-d`

目标：

- 把前端作为工作台收口，而不是带着后端瞎跑

## 8. 最终建议

如果只问一句最关键的话：

不要按“前后端 + 测试 + 文档”这种传统流水线拆。

应该按产品模块拆：

- 数据基础
- 查询智能
- 洞察预测
- 前端工作台
- CLI / MCP / 运行时

这样每个 agent 才能拥有真正连续的上下文。

## 9. 执行留痕要求

从本轮开始，以上 5 个 agent 不只是在概念上存在，而是都要有自己的独立目录。

每个 agent 至少维护以下内容：

- `orders/`：我发给该 agent 的正式指令单
- `tickets/`：该 agent 对应的正式工单
- `reviews/passed/`：审核通过记录
- `reviews/rework/`：审核不通过时的返工单与驳回记录

执行原则：

- 不允许只在聊天里说“去做某事”，必须落成文件
- 不允许只说“这里改一下”，必须形成审核意见或返工单
- agent 可以调用 sub-agent 提高效率，但只能在本模块内拆分
- 最终仍由对应 owner 对结果负责

具体编号和流程规范见 `10-Agent协作与审核规范.md`。
