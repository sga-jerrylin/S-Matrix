# 工作约定

## 文档原则

- 先写清楚事实，再写判断
- 先写清楚范围，再写执行
- 文档尽量短、准、可继续维护
- 同类信息只保留一个主要入口，避免多处冲突

## 任务原则

- 任务必须能说清目标
- 任务必须能说清边界
- 任务必须能说明完成标准
- 大任务必须拆小，不要一条任务覆盖所有事情

## 变更原则

- 改代码前先确认目的
- 改文档时同步更新相关入口
- 不做无说明的大范围重写
- 遇到方向变化，先更新文档再继续推进

## 当前阶段约定

- 当前以“项目启动 + 第一轮基建准备”为主
- 采用模块 owner 制，不按零散职能拆人
- 每个 agent 必须拥有独立文件夹
- 每张工单、指令单、审核单、返工单都必须带编号和时间
- 任何 agent 交付默认需要经过审核
- 审核不通过时必须补发返工单，不能只口头要求修改

## 文件维护建议

- `README.md` 面向外部和新接手的人
- `project-work/` 面向项目推进和内部协作
- `project-work/agents/` 面向每个 agent 的任务执行留痕
- `.plans/` 作为历史实现与分析参考，先保留，不在本轮重构

## Agent 协作约定

- 指令单负责写清“做什么、不做什么、授权到什么程度”
- 工单负责写清目标、边界、完成标准、交付物
- 审核单负责给出通过或不通过结论
- 返工单负责写清退回原因、修正范围和再次提交要求
- agent 如需使用 sub-agent，只能在自己模块内拆分，不得跨模块越权改动
- 使用 sub-agent 后，最终 owner 仍对交付质量负责
